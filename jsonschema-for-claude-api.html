00

Why schemas matter for LLMs

The model is non-deterministic. Your code isn't.

JSON Schema is a small JSON document that describes what other JSON documents are allowed to look like. Types, required keys, value ranges, enums. A contract.

You need it because your_app.process(claude_response) assumes a specific shape. If Claude returns {"age": "thirty"} when your code expects {"age": 30}, the next line crashes. The schema is how you tell Claude "here's the exact shape," and how you tell your code "here's what to verify before trusting it."

The Claude API uses JSON Schema in three places: defining tools the model can call, forcing structured output by treating the model's tool-call input as your data, and validating + retrying when the model gets the shape wrong. We'll cover all three.

The mental model. Schema = contract. Validation = enforcement of the contract. Retry = sending the contract violation back to the model so it tries again. Every production AI app you build needs all three.

Demo: Required vs optional

What "required" really means.

For objects, you list allowed keys in properties. By default every key is optional. To force a key to exist, list it in required.

Common confusion: not being in required doesn't mean the key is forbidden, only that it can be absent. If the key shows up, its value still has to match its property schema.

Schemas inside schemas. Where deep error paths come from.

Real-world data isn't flat. Properties can be objects with their own properties. Arrays can have rules about what their items look like via items.

When validation fails deep inside a nested structure, the error's path shows you exactly where. ['address', 'zip'] means "the zip field inside the address field of the root object." Array indices appear as numbers: ['books', 0, 'title'].

Demo: Structured output via tool_choice

The trick: a tool whose input is your output.

The Claude API doesn't have a separate "structured output" mode. Instead you exploit the tool-use system: define a tool whose input_schema describes the data shape you want, then force Claude to call it with tool_choice: {"type": "tool", "name": "..."}. The arguments Claude sends become your structured data.

This works because Claude is required to produce a JSON object that conforms to the tool's input_schema. You're not really executing a tool; you're using the schema enforcement as a structured output mechanism.

09

Footguns

The mistakes you'll make on your first real schema.

1. Required-everywhere causes fabrication

If you mark every field as required, Claude is contractually obligated to produce a value for each. When the source data doesn't actually contain that info, the model invents one. Make fields optional when they might genuinely be absent.

2. Nullable fields need explicit handling

"Optional" and "nullable" aren't the same. Optional means the key can be missing. Nullable means the key is present but the value is null. To allow both:

// allow string OR null (nullable)
{"type": ["string", "null"]}

// or equivalently
{"anyOf": [{"type": "string"}, {"type": "null"}]}

3. Enums are brittle. Use "other" + detail.

An enum like ["bug", "feature", "docs"] works until someone files an issue that doesn't fit. Add an "other" option plus a sibling detail field, so the model has somewhere to put unusual cases instead of cramming them into the wrong slot:

{
  "type": "object",
  "properties": {
    "category": {"enum": ["bug", "feature", "docs", "other"]},
    "category_detail": {"type": "string"}
  },
  "required": ["category"]
}

4. Over-strict patterns cause hallucination

A regex like ^[A-Z]{2}-\d{6}$ forces Claude to produce something matching that shape, even when the source text says "the invoice number is somewhere on page two." The model will invent AB-123456 rather than admit it doesn't know. Either relax the pattern, or make the field optional, or add an "unknown" escape valve.

5. Descriptions are part of the schema

Every property can have a description field. Claude reads it. A schema with rich descriptions ("ISO 8601 date like 2025-03-05, never include time") consistently outperforms a schema with the same structure but no descriptions. Treat property descriptions as part of the prompt.

The unifying principle: a schema is a prompt. Every required, enum, pattern, and description shapes the model's behavior. Restrictive schemas produce confident, wrong output. Permissive schemas with good descriptions produce honest output.

Putting it together

A complete tool, ready to paste into your app.

Here's an entity extractor that uses everything from this page: a forced tool_choice for structured output, nullable fields, an "other" enum escape valve, rich descriptions, and validation after the call.

import anthropic, jsonschema

client = anthropic.Anthropic()

EXTRACT_TOOL = {
    "name": "extract_entities",
    "description": "Extract named entities from text. Use this whenever the user provides text containing people, organizations, places, or dates that should be identified.",
    "input_schema": {
        "type": "object",
        "required": ["entities"],
        "properties": {
            "entities": {
                "type": "array",
                "items": {
                    "type": "object",
                    "required": ["text", "type"],
                    "properties": {
                        "text": {"type": "string", "description": "The entity as it appears in the source text."},
                        "type": {
                            "enum": ["person", "company", "location", "date", "other"],
                            "description": "Best-fit category. Use 'other' for anything that doesn't cleanly fit."
                        },
                        "type_detail": {
                            "type": ["string", "null"],
                            "description": "Free-text clarification when type='other'. Null otherwise."
                        }
                    }
                }
            }
        }
    },
}

resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=[EXTRACT_TOOL],
    tool_choice={"type": "tool", "name": "extract_entities"},
    messages=[{"role": "user", "content": text}],
)

# Pull the tool_use block out of the response
tool_block = next(b for b in resp.content if b.type == "tool_use")
data = tool_block.input

# Validate, even though the schema was enforced -- belt and suspenders
validator = jsonschema.Draft202012Validator(EXTRACT_TOOL["input_schema"])
errors = list(validator.iter_errors(data))
if errors:
    # Build a retry prompt with the full error list (not just one)
    retry_with_errors(text, data, errors)
else:
    return data["entities"]
This is the whole pattern. Tool with rich descriptions, forced tool_choice, validate after, retry with the full error list. Every structured-output use case you'll build follows this template.
