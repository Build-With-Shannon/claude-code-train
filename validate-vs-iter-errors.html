The myth: validate() raises the first error.

It doesn't. It raises the one best_match() picks. This page runs the real jsonschema library in your browser to prove it, and shows you exactly why title wins.

01

The schema

Three independent rules. A payload can break all three at once.

# learn_schemas.py
BOOK_SCHEMA = {
    "type": "object",
    "properties": {
        "title":  {"type": "string"},
        "year":   {"type": "integer", "minimum": 1500, "maximum": 2100},
        "rating": {"type": "integer", "minimum": 1,    "maximum": 5},
    },
    "required": ["title", "year", "rating"],
}
Why this schema matters. required sits at the root, so when it fires, the error's path is []. minimum and maximum live inside properties, so their paths are ['year'] and ['rating']. That path difference is the whole game.

04

Why title wins

best_match's relevance key, computed live for each error.

Run validation to see the relevance breakdown.

The rule: best_match picks the error with the highest (-len(path), validator not in {anyOf, oneOf}). Shorter path wins. Root-level errors beat property-level errors. Iteration order is irrelevant.
05

The receipts

Run this yourself. Verify nothing here is invented.

jsonschema.validate() source (from the library) 
# jsonschema/validators.py (simplified)
def validate(instance, schema, cls=None, *args, **kwargs):
    if cls is None:
        cls = validator_for(schema)
    cls.check_schema(schema)
    validator = cls(schema, *args, **kwargs)
    error = exceptions.best_match(validator.iter_errors(instance))
    if error is not None:
        raise error

The module-level function explicitly calls best_match. The instance method validator.validate() does NOT. That's the source of the confusion.

best_match's relevance function (from jsonschema.exceptions) 
# jsonschema/exceptions.py (simplified)
WEAK_VALIDATORS = {"anyOf", "oneOf"}

def relevance(error):
    validator = error.validator
    return (
        -len(error.path),         # shorter path wins
        validator not in WEAK_VALIDATORS,  # non-disjunction wins
    )

def best_match(errors, key=relevance):
    errors = iter(errors)
    best = next(errors, None)
    if best is None:
        return
    for error in errors:
        if key(error) > key(best):
            best = error
    return best
Falsifiable test: run this in your own terminal

If validate() really raised "the first error," this script would print the year error for both. It doesn't.

import jsonschema
from jsonschema.exceptions import best_match, relevance

BOOK_SCHEMA = {
    "type": "object",
    "properties": {
        "title":  {"type": "string"},
        "year":   {"type": "integer", "minimum": 1500, "maximum": 2100},
        "rating": {"type": "integer", "minimum": 1,    "maximum": 5},
    },
    "required": ["title", "year", "rating"],
}
BAD_BOOK = {"year": 1200, "rating": 7}

validator = jsonschema.Draft202012Validator(BOOK_SCHEMA)

# Path A: module-level — uses best_match
try:
    jsonschema.validate(BAD_BOOK, BOOK_SCHEMA)
except jsonschema.ValidationError as e:
    print("module validate()  ->", e.message)

# Path B: instance method — raises first from iter_errors
try:
    validator.validate(BAD_BOOK)
except jsonschema.ValidationError as e:
    print("instance validate()->", e.message)

# Show the relevance scores
for e in validator.iter_errors(BAD_BOOK):
    print(f"path={list(e.absolute_path)!r:15} rel={relevance(e)}")
06

The lesson

What students should walk away with.

✗ The myth

validate() raises the first error

validate()    — first error, raises. Good for
                retry loops where you want the
                model to fix one thing at a time.

Two errors in one sentence: it's not "first," and feeding errors one-at-a-time to an LLM retry loop multiplies your round trips.

✓ The reality

validate() raises one error chosen by best_match

validate()    — raises ONE error chosen by
                best_match (shortest path wins,
                not iteration order). Use for
                hard-fail boundaries: reject and stop.

iter_errors() — yields every violation.
                Use for LLM retry loops (feed
                the full list back), UI, audit logs.

is_valid()    — boolean only. Cheapest when
                you only need yes/no.
